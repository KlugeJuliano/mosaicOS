8 princípios fundacionais

⬡
Microkernel-first
Kernel mínimo: memória, scheduling, IPC, capabilities. Tudo mais em user-space.
◻
Graphical by default
Boot direto num ambiente gráfico. Não é terminal com GUI adicionado depois.
⬚
Container-native
Toda aplicação roda em contexto isolado. Inspirado em Docker, FreeBSD Jails.
◈
User-space services
Storage, network, drivers, áudio — todos fora do kernel.
≡
Declarative config
Sistema descrito em YAML/TOML. Serviços, containers, recovery — tudo declarado.
⌁
Translators (Hurd)
Recursos (cloud, DB, rede) expostos como parte da árvore de filesystem.
↺
Self-healing
Crash no compositor não derruba o sistema. Recovery é primitiva do OS.
⊕
Capability security
Serviços recebem somente o que precisam. Zero permissões implícitas.
Hard constraints — o que NÃO fazer agora
Não rodar em hardware real ainda
QEMU é o target inicial. Hardware real = problema de driver antes do tempo.
Não buscar compatibilidade POSIX completa
Linux personality é pesquisa futura, não milestone inicial.
Não implementar GPU / aceleração gráfica
Framebuffer via QEMU é suficiente até o Milestone 5.
Não construir sistema de arquivos do zero (ZFS, Btrfs)
Usar filesystem existente como base, expor via translator se necessário.
Scope constraints — foco do projeto
Cada milestone deve produzir algo que roda
Nada de design puro sem execução. Mesmo que seja um hello world em QEMU.
Documentar antes de implementar
Boot flow, service model, recovery policy — tudo documentado antes do código.
Kernel não parseia arquivos de configuração
mosaic-init é responsável por ler o system manifest e iniciar serviços.
Proteção de licença antes do primeiro código
Escolher MIT, BSD-2, Apache ou GPL antes de qualquer commit de código.
Tech constraints — decisões de stack
Microkernel base: L4Re / Fiasco.OC
Experimentar L4Re primeiro antes de considerar alternativas (seL4, Genode).
Implementações iniciais em Rust ou C
Rust para serviços de user-space. C para partes próximas ao kernel/ABI.
IPC deve ser o único canal entre serviços
Sem memória compartilhada implícita entre componentes isolados.
WASM antes de Linux personality
Ordem de complexidade crescente: native → WASM → Linux ABI.
Quality constraints — padrão mínimo
Todo serviço deve ter política de recovery declarada
restart, fallback, ou fail-fast — nunca silêncio.
Capabilities são auditáveis em todo momento
Deve existir ferramenta (mosaicctl) para inspecionar permissões de qualquer serviço.

M1
L4Re Laboratory Fase 0
Ambiente de laboratório: compilar e rodar L4Re no QEMU. Hello world funcional.
Setup do ambiente — Docker ou VM com toolchain L4Re
Build L4Re do source — compilar Fiasco.OC + L4Re userland
Boot em QEMU — script de boot funcional e reproduzível
Hello world task — primeiro programa rodando como tarefa L4Re
Documentar — passo a passo no docs/lab-setup.md
Estudar IPC — enviar mensagem entre duas tarefas via IPC L4Re
M2
mosaic-init mínimo Fase 1
Init que lê um manifest simples e inicia um serviço de user-space com logging.
Parser de manifest — ler YAML/TOML com definições de serviços
Iniciar primeiro serviço — spawnar tarefa L4Re a partir do manifest
Serviço de log básico — logd recebendo mensagens via IPC
Healthcheck simples — ping/pong para verificar se serviço está vivo
Validação de manifest — erros claros em caso de configuração inválida
M3
Service Manager Fase 1
Múltiplos serviços com rastreamento de estado, dependências e restart.
Grafo de dependências — ordenar serviços por requires: antes de iniciar
State machine de serviço — starting → running → failed → restarting
Restart automático — política restart: always / on-failure / never
mosaicctl status — listar serviços e estados
Capability assignment — passar capabilities definidas no manifest
Testes — iniciar 3+ serviços com dependências e verificar ordem correta
M4
Recovery Prototype Fase 2
Watchdog, detecção de falha, restart e logs de recovery.
Implementar watchdog — heartbeat timeout entre serviço e guardian
Detectar crash — sinal de falha via IPC ou exit code
Política de fallback — após N restarts, tentar serviço alternativo
Crash logs — registrar timestamp, motivo e histórico de restarts
Simular crash — matar processo propositalmente e observar recovery
Recovery shell — modo seguro quando serviço crítico não recupera
M5
Graphical Prototype Fase 2
Framebuffer + compositor mínimo + janela + input. Tudo em QEMU.
Display service — escrever pixels no framebuffer via QEMU VGA
Compositor mínimo — compositar buffers de janelas na tela
Protocolo GUI — definir mensagens IPC entre app e compositor
Input server — capturar teclado/mouse via QEMU
Primeira janela — app que desenha um retângulo e recebe input
Integrar recovery — crash do compositor → safe-gui mode
M6
Translator Prototype Fase 3
Virtual filesystem translator expondo um recurso como diretório.
Definir translator API — interface IPC para montar/desmontar translators
Translator simples — expor um recurso estático como diretório
Mount no manifest — configurar translator em mounts: no YAML
Lifecycle — translator iniciado pelo service manager como qualquer serviço
translator-sdk v0.1 — biblioteca para criar novos translators
M7
Container Prototype Fase 3
Manifest de container, processo isolado com capabilities, proxy GUI.
Container manifest — schema de containers: no YAML com runtime, gui, mounts, network
Processo isolado — namespace separado via capabilities L4Re
GUI proxy — app dentro do container acessa compositor via proxy
Policy engine — avaliar permissões de clipboard, screen capture, rede
mosaic-run — CLI para iniciar container a partir do manifest
M8
WASM Runtime Fase 4
Rodar módulo WASM com APIs controladas do sistema.
Escolher runtime WASM — Wasmtime ou wasm3 como base
Hospedar runtime — rodar o engine WASM como serviço user-space
Expor syscall API — interface controlada para I/O, GUI, logging
Rodar módulo hello world — WASM compilado rodando no ambiente
Sandboxing — verificar que módulo não pode escapar das capabilities
M9
Linux Personality Research Fase 4
ELF loading, syscalls básicos, tentar rodar binário Linux mínimo.
Estudar ELF — entender formato e loading de binários x86-64
Implementar write() e exit() — primeiros 2 syscalls Linux
Rodar binário estático — tiny static Linux binary imprimindo texto
Documentar aprendizados — quais syscalls são viáveis, quais são complexos demais
Avaliar continuidade — decidir se Linux personality é roadmap de longo prazo
